# 前端重构过程与方法论

本文档记录 PPTTalK 前端从 **Uni-App H5 编译产物** 迁移到 **Vue 3 + Vite + TypeScript**（`frontend-new/`）的完整过程，包括决策依据、信息来源、实施步骤和遇到的问题。可与 [前端重写与API对照内容.md](前端重写与API对照内容.md)、[修缮与功能总结.md](修缮与功能总结.md) 对照阅读。

---

## 一、重构前的状况

### 1.1 旧前端的形态

| 项目 | 现状 |
|------|------|
| 框架 | Uni-App（跨端：H5 / 小程序 / App） |
| 源码 | **不在仓库中**，仅存编译后的 minified JS 产物 |
| 部署位置 | `/home/ubuntu/web/`，由 Nginx 在端口 5173 提供静态文件服务 |
| 已有补丁 | `frontend/assets/pptalk-patch.js`，通过 MutationObserver 在运行时注入 UI（文案风格选择、背景音乐、情感控制等） |

### 1.2 编译产物构成（3 个页面）

| 页面 | 编译产物 | 大小 | 功能 |
|------|----------|------|------|
| 首页 | `pages-index-index.EbKpVKa8.js` | 4.6 KB | 欢迎 / 入口，几乎是静态页 |
| 上传 / 生成 | `pages-file_upload-file_upload.js` | 70 KB | 核心：上传 → 预览 → 编辑讲稿 → 配置 → 生成 → 结果 |
| 视频列表 | `pages-video_list-video_list.js` | 523 KB | 其中绝大部分为 hls.js 库体积，业务逻辑很少 |

### 1.3 存在的问题

1. **源码缺失**：只能在 minified JS 上打补丁，不可维护
2. **Uni-App 限制**：跨端抽象层导致不能直接使用 `EventSource`、标准 DOM API；网络请求被迫走 `uni.request`，是轮询问题的根源之一
3. **补丁模式脆弱**：`pptalk-patch.js` 依赖 DOM 结构猜测，任何 UI 变动都可能导致注入失败
4. **IP 硬编码**：编译产物中 API 地址写死为旧服务器公网 IP，迁机后必须手动搜索替换 minified JS

## 二、决策：重做 vs. 源码改造

### 2.1 重做的理由

| 维度 | 分析 |
|------|------|
| 规模评估 | 整个前端仅 **3 个页面**、调用 **7 个后端 API**，重写工作量可控 |
| 框架瘦身 | 项目只需 Web 端，Uni-App 的跨端能力完全多余，Vue 3 + Vite 更轻量直接 |
| 可维护性 | 全新源码 + TypeScript 类型安全，远优于在 minified JS 上打补丁 |
| SSE 控制 | 可以完全掌控连接生命周期（`fetch` + `ReadableStream`），解决 Cursor 远程环境下 `EventSource` 自动重连触发 `vscode-remote://` 弹窗的问题 |

### 2.2 保留的部分

- **后端核心不动**：`digital_human_interface` 的流水线编排（TTS → Wav2Lip → 合并 → 切片）完全不改
- **视觉风格延续**：重写页面保留旧版 UI 外观

## 四、实施步骤

### 4.1 后端适配（不改核心逻辑）

| 改动 | 文件 | 说明 |
|------|------|------|
| CORS 白名单 | `config/settings.py`、`main.py` | 从硬编码 IP 改为 `.env` 环境变量驱动 |
| SSE 端点拆分 | `routers/my_digital_human.py` | 原 POST 同时创建+推流 → 拆为 `POST /tasks`（创建任务）+ `GET /tasks/{id}/stream`（SSE 订阅） |
| BGM 路径 | `routers/files.py` | `upload_bgm` 返回值从绝对路径改为相对 URL |
| 响应格式 | 多处 | 统一为 `{ code, message, data }` 信封格式 |

### 4.2 前端搭建

```
frontend-new/
├── package.json          # Vue 3 + Vite + TypeScript + hls.js
├── vite.config.ts        # 代理、构建配置
├── tsconfig.json
├── .env                  # VITE_API_BASE
├── src/
│   ├── api/
│   │   ├── client.ts          # apiGet / apiPost / apiUpload / apiDelete + staticUrl / sseUrl
│   │   ├── digitalHuman.ts    # 数字人 CRUD + 任务创建
│   │   ├── files.ts           # PPT 上传、文案生成、BGM 上传
│   │   ├── conversion.ts      # 转换状态轮询
│   │   └── video.ts           # 视频列表
│   ├── composables/
│   │   └── useSSE.ts          # fetch + ReadableStream 手动解析 SSE 协议
│   ├── pages/
│   │   ├── IndexPage.vue      # 首页
│   │   ├── UploadPage.vue     # 核心：上传 → 生成 全流程
│   │   └── VideoListPage.vue  # 视频列表 + HLS 播放
│   ├── components/
│   │   └── GenerationProgress.vue  # 生成进度展示
│   ├── router/
│   │   └── index.ts           # Hash 模式路由
│   └── assets/styles/
│       └── global.css         # 全局样式变量
```

### 4.3 关键技术决策

| 决策 | 选择 | 理由 |
|------|------|------|
| SSE 实现 | `fetch` + `ReadableStream` 手动解析 | 避免 `EventSource` 在 Cursor 远程端口转发下自动重连触发 `vscode-remote://` 弹窗；可完全控制连接生命周期 |
| 路由模式 | Hash 模式 (`/#/`) | 兼容 Nginx 静态托管，无需配置 `try_files` 回退 |
| UI 库 | 无（纯 CSS） | 3 个页面体量不值得引入组件库；保持与旧版视觉一致 |
| 视频播放 | hls.js | 后端输出 HLS 切片（`.m3u8` + `.ts`），与旧版一致 |
| HTTP 客户端 | 原生 `fetch` | 无需 axios 等额外依赖 |

### 4.4 部署对接

1. `cd frontend-new && npm run build` 生成 `dist/` 产物
2. 将 `dist/` 内容复制到仓库根目录 `frontend/`（即 Nginx 静态根）
3. Nginx 重载：`sudo nginx -s reload`

## 五、重构后的 Bug 修复

以下问题在重构上线后陆续发现并修复（详见 [修缮与功能总结.md](修缮与功能总结.md) 第六章）：

| 编号 | 问题 | 根因 | 修复方式 |
|------|------|------|----------|
| 6.1.1 | TTS 显存持续堆积 | PyTorch CUDA 缓存未释放 | 推理后 `gc.collect()` + `torch.cuda.empty_cache()`；FastAPI 中间件兜底 |
| 6.1.2 | SSE 进度日志空行 | 心跳注释行（`: heartbeat`）未过滤 | `useSSE.ts` 识别注释行跳过 + 渲染层二次过滤 |
| 6.1.3 | 计时器刷新后重置 | `localStorage.startTime` 被误清 + 已完成任务无计时器 | 优先级恢复链：localStorage → 后端 `created_at`；已完成时冻结显示 |
| 6.1.4 | 数字人位置全映射左下 | `location_dir` 键为中文，新前端发英文值 | 补充英文键映射，兼容新旧前端 |
| 6.1.5 | `vscode-remote://` 弹窗 | `EventSource` 自动重连被 Cursor 代理拦截 | 已在重构中用 `fetch` + `ReadableStream` 替代 |
| 6.1.6 | 进度条先升后降 | 阶段 1 硬编码 `progress=10` 高于后续按时间分摊的累计值 | 引入 `STAGE1_MIN_PROGRESS` 锚定 + 后续按比例分配 + 完成时强制 100% |
| 6.1.7 | SSE 进度整段重复两遍 | 同一事件同时进 `task_event_history` 与队列，订阅端先重放 history 再读队列导致双份 | `stream_generation_task` 重放后跳过队列中与 history 等长的前缀 |

上线后另有 **首页文案** 调整为「AI PPT」系列标题、**步骤 1 欢迎字幕**（`welcome_text`）由 `WelcomeTextConfig.vue` 补全，详见 [修缮与功能总结.md](修缮与功能总结.md) 6.2.2、6.2.3。

## 六、经验总结

### 6.1 可复用的方法

1. **逆向先行**：在无源码/无文档时，从编译产物和后端路由两端夹逼，可以高效还原出完整的接口契约
2. **后端先改、前端后建**：先确保 API 层干净可测，再搭前端调用，减少联调摩擦
3. **SSE 手动解析**：在非标准浏览器环境（IDE 内嵌、端口转发代理）下，`fetch` + `ReadableStream` 比 `EventSource` 更可控

### 6.2 待改进

| 项目 | 现状 | 建议 |
|------|------|------|
| API 类型共享 | 前后端各自定义 TypeScript 接口 / Pydantic 模型 | 可引入 OpenAPI → TypeScript codegen 自动生成 |
| 环境配置 | `.env` 手动维护 | 可通过 `setup_env.sh` 自动生成前后端 `.env` |
| 端到端测试 | 无 | 可用 Playwright 覆盖核心流程（上传 → 生成 → 播放） |
