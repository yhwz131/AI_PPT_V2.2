# PPTTalK 修缮与功能总结

本文档归纳 PPTTalK（AI 数字人 PPT 讲解视频流水线）在本轮迭代中的 **Bug 修复**、**新增功能**、**优化与运维建议**（含 IP 变更策略），以及 **项目创新点与亮点**。可与 [文档索引.md](文档索引.md)、架构与数据流文档（`jna*.md`）及 [`各服务配置与常见问题说明.md`](各服务配置与常见问题说明.md) 对照阅读。

## 一、Bug 修复（详述）

### 1. 服务与部署

| 问题 | 现象或根因 | 处理要点 |
|------|------------|----------|
| Wav2Lip 路径错误 | `start_all.sh` 指向不存在的目录，服务无法启动 | 与仓库实际路径对齐（如 `wav2lip_workspce/lx/测试`） |
| 内外网 IP 硬编码 | 旧公网/内网地址写死在多处，换机或迁云后请求失败 | **服务间**统一 `127.0.0.1`；**浏览器访问**通过 Nginx 或环境注入的 `baseURL`，避免在 Python 里写死公网 IP |
| `start_all.sh` 启停异常 | `local` 误用、`$0` 解析错误；TTS 子进程残留 | 修正 shell；`setsid` + 对进程组 `kill -TERM`，确保 vLLM 等子进程退出 |
|日志不实时 | stdout 缓冲 | `export PYTHONUNBUFFERED=1` |

### 2. 文案与编排

| 问题 | 处理要点 |
|------|----------|
| 手动编辑讲稿后解析失败 | 讲稿为 HTML 时，后端增加 BeautifulSoup 等兜底解析，避免单一格式假设 |
| 大 PPT 预览超时 | 后端 LibreOffice 转换时间可能超过原前端轮询窗口；前端延长轮询与上传超时 |

### 3. GPU 与生成稳定性

| 问题 | 处理要点 |
|------|----------|
| TTS / vLLM OOM | 调整 `gpu_memory_utilization`、`qwenemo_gpu_memory_utilization`、`is_fp16`；规范进程退出释放显存 |
| 多路 Wav2Lip 并行 OOM | 编排层改为**顺序**生成单页视频，避免同卡多路推理 |
| PPT 转 PDF任务等待不足 | `poll_task_status_async` 等提高 `max_attempts`、缩短 `interval`，兼顾长转换与响应速度 |

### 4. Wav2Lip 与依赖

| 问题 | 处理要点 |
|------|----------|
| MediaPipe 不可用 | Python 3.8 需匹配 `mediapipe` 版本（如 0.10.9），避免新版权限/语法不兼容 |
| 「无数字人」模板失败 | `location_dir` 缺键导致 KeyError；且 API 强制要求 `face_video` | 增加 `"无数字人": "none"`；`face_video` 可选；流水线跳过 Wav2Lip 人脸分支，仅背景+音频+字幕 |

### 5. 前端（Uni-App 编译产物 + 补丁）

| 问题 | 处理要点 |
|------|----------|
| 状态历史不自动滚到底 | `scroll-top` 恒为同一值时 Vue 不更新，改为 999999/999998 交替 |
| 新任务开始旧结果仍高亮 | 在展示新任务 loading **之前**调用状态重置（如 `Ka()`） |
| 视频列表进度条无法拖拽 | `progress-track` 增加 click/touch seek；`getBoundingClientRect` 使用真实 DOM（`$el`） |
| 情感向量超约束仍可下一步 | 设置 `window.__pptalk_vec_valid`；编译 JS 在步骤切换前校验 |
| 进入下一步后难以返回 | 步骤条在允许时**可点击回退**；保留底部「返回」；补丁 CSS 提示可点击步骤 |
| 浏览器缓存旧 API 地址 | Nginx 缓存头与强制刷新策略需配合（见第三章） |

### 6. 文案篇幅与风格一致性

三档风格（简略/普通/专业）的每页目标字数在 `get_voice_content.py` 的 `STYLE_PRESETS` 与前端 `pptalk-patch.js` 提示文案中应**保持一致**；修改时两处同步。

## 二、新增功能（详述）

### 1. 讲稿文案风格（三档）

- **参数链路**：前端 `formData.style` → `digital_human_interface` `/files/voice_over_script_generation` → PaddleOCR `POST /parse`（`style` 表单字段）→ `get_ppt_voice_content(md, style=...)`。
- **预设**：`brief` / `normal` / `professional` 对应不同 `tokens` 上限与风格描述（见 `paddleocr/until/get_voice_content.py`）。

### 2. 模板页：欢迎词与背景音乐

- **欢迎词**：SSE JSON 增加 `welcome_text`，透传至 Wav2Lip 合成（标题动画等仍使用每页 `topic_name` 等业务字段）。
- **背景音乐**：`bgm_mode`: `default` | `custom` | `none`；自定义先 `POST /files/upload_bgm` 得 `bgm_path`；Wav2Lip 表单 `bgm_enabled`、`bgm_path`。

### 3. TTS 情感控制

- **自然语调**：`emo_control_method=0`（默认）。
- **情感向量**：`emo_control_method=2` + `emo_vec`（JSON 数组字符串）；前端约束：**每维 [0, 1.2]，八维之和 ≤ 1.5**，不满足禁止进入下一步。
- **情感文本**：`emo_control_method=3` + `emo_text`。
- **后端**：`audio_generation` 将字段透传至 `http://127.0.0.1:6006/batch_tts_url`。

### 4. 前端补丁机制

- **`frontend/assets/pptalk-patch.js`**：`MutationObserver` + 防抖，在步骤 DOM 重建后注入卡片式 UI；读写 `window.__pptalk_*`。
- **`frontend/index.html`**：在 `#app` 之后引入补丁（避免阻塞主包）。
- **编译包**：`pages-file_upload-*.js` 中口播请求与 SSE body 读取上述全局变量。

### 5. 「无数字人」成片

- 编排层不传 `face_video`；Wav2Lip `_process_parallel` 跳过 wav2lip 任务；`generate_video` 直接将背景视频与口播音轨合并，再走字幕与音效。

### 6. 步骤导航体验

- 顶栏步骤在「已完成/可回退」状态下可点击跳转；与底部「返回」互补。

## 三、优化与运维建议

### 3.1 IP / 域名变更时如何保证地址一致

**原则：区分三类地址，不要混用。**

1. **进程间调用（推荐全部写死为回环地址）**  
   - Python 服务调用 PaddleOCR、TTS、Wav2Lip：使用 `http://127.0.0.1:端口`。  
   - 优点：不随公网 IP、域名变更；不依赖本机 hostname 解析。

2. **浏览器 / H5 客户端访问 API**  
   - 由 **相对路径**（同域部署）或 **单一配置源** 提供 `baseURL`（如构建时注入、或 Nginx 反代后前端只写 `/api`）。  
   - 避免在多个 JS 包里复制粘贴完整 `http://公网IP:端口`。

3. **对外暴露**  
   - 公网 IP 或域名只出现在：**Nginx `server_name` / `proxy_pass`、DNS、防火墙白名单**，不要散落进各微服务业务代码。

**换 IP 时的检查清单（建议做成发布 checklist）：**

- [ ] Nginx：upstream 或 `proxy_pass` 是否仍指向本机 `127.0.0.1:9088`（或其它内部端口）。  
- [ ] 前端：`$baseURL` 或等价变量是否来自环境或相对路径；清理 CDN/浏览器强缓存。  
- [ ] `digital_human_interface/static/data/basic_information.json` 等静态 JSON 中的绝对 URL。  
- [ ] `.env`、README 示例、旧文档中的示例 IP（不影响运行但避免误导）。  
- [ ] CORS：若前端与 API 不同域，更新 `allow_origins`。  
- [ ] 防火墙 / 安全组：仅开放80/443（或业务端口），内网服务端口不对外。

### 3.2 配置集中化（中长期）

- 将「对外基础 URL」「各微服务 URL」收敛到 **环境变量** 或 **单一 `config.yaml`**，由 `digital_human_interface` 读取；子服务只保留端口监听与模型路径。  
- 前端 Uni-App 若可重新构建，用 **编译期 env** 注入 `VITE_*`，避免长期直接改 minified JS。

### 3.3 可观测性与稳定性

- 统一日志目录（如 `logs/`），配合 `start_all.sh status` 巡检。  
- 对 TTS、Wav2Lip 等长任务保留 **任务 ID** 与 **阶段 SSE事件**，便于定位卡在音频还是视频。  
- GPU 服务器建议记录 `nvidia-smi` 采样或简单告警，避免 silent OOM。

### 3.4 安全

- API Key、Token 仅存环境变量；`.env`、`*token*` 入 `.gitignore`（见仓库提交规范）。  
- 上传目录与执行路径隔离，防止路径穿越（TTS 侧已有 `safe_join_ref_audio` 一类实践，可推广思路）。

### 3.5 性能（在稳定前提下）

- TTS：在显存允许范围内再调 `gpu_memory_utilization`；关注 vLLM 子进程是否随主进程退出。  
- Wav2Lip：已存在人脸检测缓存等优化；多页视频仍以顺序为主，避免盲目并行。  
- PaddleOCR：大 PDF 可考虑限流或队列，防止与 Wav2Lip 争用同卡。

## 四、项目整体创新点与亮点

1. **端到端「PPT → 讲稿 → 语音 → 数字人/纯幻灯片视频」闭环**  
   文档解析、文案生成、音色克隆式 TTS、唇形同步与合成、合并输出在同一套编排下完成，降低多工具拼接成本。

2. **多模态与 LLM 协同**  
   PaddleOCR-VL 做版式与内容理解，DeepSeek（LangChain）生成结构化口播，适合企业内训、招生宣传等场景的快速成片。

3. **IndexTTS + vLLM 的工程化落地**  
   批量 `batch_tts_url`、情感多模式（含向量与文本驱动），在延迟与吞吐之间可做显存参数权衡。

4. **可编排的呈现形态**  
   浮层数字人位置模板 + **无数字人**模式，同一套音频与字幕链路可输出「有人讲解」或「纯课件+配音」。

5. **实时进度与体验**  
   SSE 推送阶段状态；前端补丁在无法改 Vue 源码的情况下扩展风格、BGM、情感与校验，兼顾交付速度与可维护性风险（中长期建议源码级集成）。

6. **运维向改进**  
   一键 `start_all.sh`、进程组级停止缓解 GPU 泄漏、内部回环地址降低环境耦合。

## 六、2026-04-17 更新日志

### 6.1 Bug 修复

#### 6.1.1 TTS 服务显存持续堆积（GPU 内存泄漏）

| 项目 | 内容 |
|------|------|
| **现象** | TTS 服务（IndexTTS）运行 ~5 小时后，主进程 GPU 显存从 7.3GB 增长到 16.3GB，GPU 1 总占用达 86% |
| **根因** | PyTorch 推理过程中产生的中间 CUDA 张量被缓存分配器持有，长时间运行后碎片堆积，`torch.cuda.empty_cache()` 从未被调用 |
| **修复** | 1. `indextts/infer_vllm_v2.py`：在 `infer()` 方法末尾增加 `finally` 块，显式 `del wavs, wav` → `gc.collect()` → `torch.cuda.empty_cache()`<br>2. `server/tts_server_v2_batch.py`：新增 `CudaCacheCleanupMiddleware`（FastAPI 中间件），仅在 `/tts_url`、`/batch_tts_url` 请求完成后触发 GC+缓存清理作为兜底 |
| **效果** | 10 次连续推理后显存稳定在 8.5~9.1GB 范围内波动，不再单调增长 |

#### 6.1.2 SSE 进度日志出现空行

| 项目 | 内容 |
|------|------|
| **现象** | 生成进度日志中出现多个只有圆点没有文字的空行，刷新后消失 |
| **根因** | 后端每 10 秒发送 SSE 心跳消息 `: heartbeat\n\n`（SSE 注释格式）。原生 `EventSource` 自动忽略注释行，但替换后的 fetch-based SSE 解析器未处理以 `:` 开头的注释行，将心跳解析为 `{ type: 'message', data: '' }` 空事件 |
| **修复** | 1. `useSSE.ts`：SSE 解析器增加对注释行（`:` 开头）的识别，纯注释块直接跳过<br>2. `GenerationProgress.vue`：渲染层增加 `v-if="formatEvent(evt)"` 二次过滤 |

#### 6.1.3 生成页计时器刷新后重置

| 项目 | 内容 |
|------|------|
| **现象** | 页面刷新后计时器从 00:00 重新开始，而非恢复到正确的已用时间 |
| **根因** | 1. 任务已完成时刷新：`completed` 分支未设置计时器，直接显示 00:00<br>2. SSE 连接短暂断开时 `clearTaskStorage()` 误清了 `localStorage` 中的 `startTime`，下次刷新后无法恢复 |
| **修复** | 1. `checkTaskExists` 增加返回后端 `created_at` 时间戳<br>2. 计时器恢复优先级：`localStorage.startTime` → 后端 `created_at * 1000`（秒→毫秒）<br>3. 已完成任务刷新时先 `startFrom` + `stop` 冻结显示总耗时<br>4. SSE 断连不再调用 `clearTaskStorage()`，仅任务真正结束时清除 |

#### 6.1.4 数字人位置模板全部映射为左下

| 项目 | 内容 |
|------|------|
| **现象** | 无论选择左上/右下/右上/居中，生成的视频中数字人始终出现在左下角 |
| **根因** | 后端 `location_dir` 字典的键是旧前端的中文名（如 `"浮层-左下"`），新前端发送的是英文值（如 `"bottom-left"`），`.get()` 查找全部未命中，回退到默认值 `"bottom-left"` |
| **修复** | `location_dir` 补充英文键映射（`"bottom-left": "bottom-left"` 等），兼容新旧两套前端；wav2lip 服务的 `overlay_digital_human` 本身已支持全部 5 个位置 |

#### 6.1.5 Cursor 远程环境 vscode-remote 弹窗

| 项目 | 内容 |
|------|------|
| **现象** | 生成过程中浏览器弹出 Windows 系统对话框"需要使用新应用以打开此 vscode-remote 链接" |
| **根因** | 原生 `EventSource` 在 SSE 连接断开后自动重连，Cursor 远程端口转发代理拦截重连请求并重定向到 `vscode-remote://` 协议 |
| **修复** | `useSSE.ts` 从原生 `EventSource` 重写为 `fetch` + `ReadableStream` 手动解析 SSE 协议，完全控制连接生命周期，不再自动重连 |

#### 6.1.6 生成任务总进度条先升后降（0%→5%→10% 再回到约 1%）

| 项目 | 内容 |
|------|------|
| **现象** | 前端总进度条在解析讲稿阶段先到约 5%、10%，进入后续阶段后突然回落到 1%～3% 再继续爬升，观感上前后不连贯 |
| **根因** | `my_digital_human.py` 中阶段 1 前几条 SSE 使用硬编码 `progress`（如 5、10），而阶段 1 结束后的整体进度改用「按预估时间分摊」的 `stage1_progress = int((stage1_time/total_time)*100)`。当 `total_time` 较大时，`stage1_progress` 往往远小于 10，累计进度从 10 跳回 3 左右，造成回退 |
| **修复** | 1. 阶段 1 内中间步骤改为较低且单调递增的占位（如 0→3→6），避免过早拉高总进度<br>2. 引入 `STAGE1_MIN_PROGRESS = 10`：阶段 1 完成后总进度锚定为 10%，阶段 2～7 共享剩余 90%，按各阶段预估时间比例分配，保证 `accumulated_progress` 全程非递减<br>3. 切片成功结束时将总进度置为 100%，避免各阶段 `int()` 截断导致最终不足 100% |

#### 6.1.7 SSE 进度日志整段重复两遍

| 项目 | 内容 |
|------|------|
| **现象** | 生成过程中进度区出现两段完全相同的日志（从「已连接」到「正在处理音频生成…」等整段重复） |
| **根因** | `POST /my_digital_human/tasks` 创建任务后，后台将每个事件同时写入 `task_event_history` 与 `asyncio.Queue`。客户端随后 `GET .../tasks/{task_id}/stream` 时，服务端先 **重放** `history` 中已有事件，再从 **queue** 读取；queue 中尚未被消费的事件与 history 内容相同，导致同一批事件被推送两次 |
| **修复** | `stream_generation_task`（`my_digital_human.py`）在重放 history 后，从 queue 读取时跳过与已重放条数相同的前 N 个事件，仅转发之后新产生的事件 |
