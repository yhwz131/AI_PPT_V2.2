# 各服务配置文件与常见问题说明

## 关于文件名前缀「jna」

本仓库中部分技术文档以 **`jna` 开头**（例如 `jnaaippt_architecture_v1.md`、`jnadigital_human_data_flow.md`）。**仓库内未对「jna」给出权威全称或英文展开**，可视为**项目组内部用于归档「架构 / 数据流 / 服务说明」等技术文档的文件名前缀**。与产品名 **AIPPT** 连写时常见形式为 **`jna` + `aippt`**，例如 `jnaaippt_architecture_v1.md` 表示与 AIPPT 相关的系统架构文档（v1）。若需对外统一说明，可在项目 README 或 `docs` 索引中自行定义缩写含义。

**说明：** 本文已采用**中文文件名**；其余 `jna*.md` 仍为历史沿用的英文文件名，交叉引用未改动的文档时仍使用原文件名。

本文汇总 PPTTalK 仓库内 **各子服务的配置入口**、**环境变量习惯**，以及部署与联调时 **容易出错或仍需人工注意** 的点。可与 [`jnaaippt_architecture_v1.md`](jnaaippt_architecture_v1.md)、[`jnadigital_human_data_flow.md`](jnadigital_human_data_flow.md)、[`修缮与功能总结.md`](修缮与功能总结.md) 对照阅读。

## 1. 总览

| 服务/组件 | 典型端口 | 主要配置文件或目录 | 启动入口（见 `start_all.sh`） |
|-----------|----------|-------------------|------------------------------|
| PaddleOCR | 8802 | `paddleocr/.env`，`api_paddleocr_vl_*.py` | `python api_paddleocr_vl_ai_ppt.py` |
| IndexTTS | 6006 | `index-tts-vllm/server/.env` | `python server/tts_server_v2_batch.py` … |
| Wav2Lip | 5000 | `wav2lip_workspce/lx/测试/config.py` | `python main.py` |
| digital_human_interface | 9088 | `digital_human_interface/.env`，`config/settings.py` | `python main.py` |
| 新前端（Vite） | 5173（dev） | `frontend-new/.env`、`.env.production` | `vite` / 静态部署 |
| 全栈编排 | — | 根目录 `start_all.sh`、`setup_env.sh` | 管理多进程与端口 |

端口以 **`start_all.sh` 中 `SERVICES` 定义** 为准；若某服务单独改端口，**必须同步** 调用方（尤其是 `digital_human_interface` 内写死的 URL，见下文）。

## 2. digital_human_interface（核心业务 API）

### 2.1 配置文件位置

| 路径 | 作用 |
|------|------|
| `digital_human_interface/config/settings.py` | Pydantic `Settings`：目录、`HOST`/`PORT`、`CORS_ORIGINS`、`LIBREOFFICE_PATH`、`FFMPEG` 等 |
| `digital_human_interface/config/config.py` | `get_settings()` 单例加载 |
| `digital_human_interface/.env` | 运行时覆盖（**不提交敏感内容到公开仓库**）；`settings.py` 中 `env_file = ".env"` |
| `digital_human_interface/.env.example` | 示例（当前偏精简，与完整部署项可能不一致，以 `settings.py` 为准） |

### 2.2 常见问题与风险

1. **子服务 URL 未走 Settings**  
   `routers/files.py` 中口播解析、`routers/my_digital_human.py` 中 TTS / Wav2Lip 地址仍多为 **代码内写死的 `http://127.0.0.1:端口/...`**。换端口或跨机部署时需改源码或后续抽象为环境变量。

2. **LibreOffice**  
   `LIBREOFFICE_PATH` 默认偏向 Windows 路径；Linux 下需在 `.env` 中设为本机 `soffice` 实际路径，否则 PPT→PDF 失败。

3. **CORS**  
   `CORS_ORIGINS` 需在 `.env` 中列出浏览器真实访问来源（含协议、域名、端口）；内网 IP 变更后旧条目会残留（参见现有 `.env` 中多地址并列的写法）。

4. **管理控制台**  
   `main.py` 中管理员账号/口令为硬编码，生产环境必须修改并限制网络访问（详见 [`jnadigital_human_console.md`](jnadigital_human_console.md)）。

## 3. PaddleOCR（解析 / 口播文案等）

### 3.1 配置文件位置

| 路径 | 作用 |
|------|------|
| `paddleocr/.env` | 模型目录 `VL_MODEL_DIR`、`LAYOUT_MODEL_DIR`、`UPLOAD_BASE_DIR`、`UPLOAD_BASE_URL` 等（**强依赖本机路径与部署 IP**） |
| `paddleocr/api_paddleocr_vl_ai_ppt.py` | 服务端口（可用环境变量）、**CORS 允许列表**（代码内多行 URL） |
| `paddleocr/api_paddleocr_vl_mineru.py` | 另一入口，端口等默认值 |
| `paddleocr/get_api_server_ai_ppt.py` | 客户端默认 `API_URL`（调用本服务 OCR） |

### 3.2 常见问题与风险

1. **绝对路径与 IP**  
   `.env` 中常见 `/home/...` 与公网/内网 IP；换机器或迁云后必须整体替换，否则模型找不到或回传 URL 错误。

2. **CORS 写死在代码里**  
   新增前端域名或端口时，除 `digital_human_interface` 的 CORS 外，还需改 **Paddle 服务** 的允许列表（若浏览器直连 Paddle）。

3. **端口与 digital_human_interface 不一致**  
   业务默认认为解析在 **8802**（`files.py` 写死）；若 Paddle 改端口，必须同步改调用 URL。

## 4. IndexTTS（语音合成）

### 4.1 配置文件位置

| 路径 | 作用 |
|------|------|
| `index-tts-vllm/server/.env` | `HOST`、`PORT`（常用 6006）、`MODEL_DIR`、各类 GPU/显存参数、`REF_AUDIO_ROOT_PATH`、`DEFAULT_OUTPUT_PATH`、`UPLOAD_TEMP_PATH` 等 |
| `index-tts-vllm/server/tts_server_v2_batch.py` | 入口脚本；`PORT` 等可被命令行与环境变量覆盖 |
| `index-tts-vllm/.env.example` | 示例（若存在） |

### 4.2 常见问题与风险

1. **绝对路径**  
   模型与输出目录多为本机绝对路径；仓库克隆到新目录后必须改 `.env`。

2. **显存与批处理参数**  
   `GPU_MEMORY_UTILIZATION`、`IS_FP16` 等与机器相关；OOM 时需调低，与 `start_all.sh` 里命令行参数保持一致（脚本中的 `--gpu_memory_utilization` 等会覆盖部分行为，以实际启动命令为准）。

3. **端口与调用方**  
   `digital_human_interface` 中 TTS 默认指向 **6006**；改端口需同步 `my_digital_human.py` 等处。

4. **示例脚本**  
   `api_example.py`、`test/simple_test.py` 等仍可能写死 `6006` 或 `localhost`，仅作本地测试，不参与生产链路时需留意勿误用。

5. **`start_all.sh` 与 `server/.env` 谁生效？**  
   `start_all.sh` 中 TTS 启动命令若带有 **`--gpu_memory_utilization`、`--qwenemo_gpu_memory_utilization`、`--is_fp16`** 等参数，则 **命令行优先**；未在命令行出现的项仍可用 **`index-tts-vllm/server/.env`** 中的 `PORT`、`MODEL_DIR` 等。仅执行 `python server/tts_server_v2_batch.py` 且不传参时，主要依赖 `.env` 与代码默认值。

## 5. Wav2Lip（口型 / 生成接口）

### 5.1 配置文件位置

| 路径 | 作用 |
|------|------|
| `wav2lip_workspce/lx/测试/config.py` | `PORT`（默认 5000，可由环境变量覆盖）、`host_ip` 等 |
| `wav2lip_workspce/lx/digital_human_project/config.py` | 镜像/备份目录下同类配置 |
| `wav2lip_workspce/lx/测试/main.py` | 应用入口 |

当前目录 **未必** 提供与 TTS 同级的 `.env` 模板，端口常依赖 **`start_all.sh` 启动环境** 或手动 `export`。

### 5.2 常见问题与风险

1. **与编排脚本的工作目录**  
   `start_all.sh` 固定从 `wav2lip_workspce/lx/测试` 启动；若实际工程只在 `digital_human_project` 下维护，需自行统一路径，避免双份逻辑漂移。

2. **端口 5000**  
   `digital_human_interface` 写死 Wav2Lip 为 **5000**；变更端口必须改调用方。

3. **资源路径**  
   模型、音效等路径若在代码或配置中写死，换机需逐项检查（历史上曾做过相对仓库根的路径整理，以当前代码为准）。

4. **音效目录：`lx/测试` 与 `digital_human_project` 两套路径**  
   `config.py` 以 **`__file__` 所在目录** 为 `_base`，默认音效在 **`{_base}/assets/sfx/`** 下三个文件名（完成提示音 / 转场 / BGM）。**`start_all.sh` 从 `lx/测试` 启动**，因此运行中的进程读取的是 **`wav2lip_workspce/lx/测试/assets/sfx/`**。若只把音频放在 **`digital_human_project/assets/sfx/`**，而服务仍从 **`测试`** 目录启动，会出现「已拷贝文件但仍引用失败」。解决办法：在 **`测试/assets/sfx/`** 放置同名文件，或设置环境变量 **`SFX_NOTIFICATION` / `SFX_TRANSITION` / `SFX_BGM`** 指向实际绝对路径。

5. **`image_output/` 残留**  
   Wav2Lip 单页合成结果默认落在 **`lx/测试/image_output/`**（或配置中的 `output_dir`）。**控制台「视频管理」删除**只清理 **`digital_human_interface/static/`** 下与 `basic_information.json` 关联的路径，**不会**删除 Wav2Lip 目录中的 `image_output`；需运维侧单独清理或后续加自动化策略。

## 6. 前端（frontend-new）

### 6.1 配置文件位置

| 路径 | 作用 |
|------|------|
| `frontend-new/.env` | 开发常用 `VITE_API_BASE=http://127.0.0.1:9088` |
| `frontend-new/.env.production` | 生产可 **`VITE_API_BASE=` 空字符串**，与 API 同源部署 |
| `frontend-new/vite.config.ts` | 构建与开发服务器 |

### 6.2 常见问题与风险

1. **API 基地址**  
   跨域或前后端分离时，`VITE_API_BASE` 必须指向可访问的后端；混用 HTTP/HTTPS 时浏览器会拦截请求。

2. **缓存旧环境变量**  
   修改 `.env` 后需重启 `vite`  dev server 并重新构建生产包。

## 7. 其他前端目录（new_frontend）与部署目录（frontend）

| 路径 | 作用 |
|------|------|
| `new_frontend/` | 另一套 Vue + Vite 工程（如 Element Plus），**不是** `setup_env.sh` 默认构建源 |
| `new_frontend/vite.config.ts` | 开发代理 `proxy` 默认指向 `http://127.0.0.1:9088` |
| **`frontend/`** | **静态部署根目录**（由 **`frontend-new`** `vite build` 后拷贝生成），Nginx `root` 常指向此处 |

**主线**：开发/构建用 **`frontend-new`** → 产物进 **`frontend/`**。**`new_frontend`** 仅在与团队约定时使用，避免与 `frontend-new` 混淆。

## 8. 仓库级脚本（与配置的关系）

| 文件 | 作用 |
|------|------|
| `start_all.sh` | 定义各服务 **conda 环境、工作目录、启动命令、端口**；控制台启停子进程也依赖其中的 `start_all.sh` 路径 |
| `setup_env.sh` | 一键部署时生成/提示 **conda、路径、部分 env 片段**（如 `CORS_ORIGINS`、`UPLOAD_BASE_URL`）；内容随机器变化 |

**注意：** `start_all.sh` 内 **端口与命令写死在 `SERVICES` 数组**；与 `.env` 不一致时，以**实际监听端口**为准并同步调用方。

## 9. 跨服务共性问题（检查清单）

| 现象 | 可能原因 |
|------|----------|
| 口播生成失败 | Paddle 未起、端口非 8802、或 `files.py` URL 未改 |
| TTS 无响应 | IndexTTS 未起、6006 未监听、显存不足崩溃 |
| 数字人视频卡住 | Wav2Lip 未起、5000 不通、GPU OOM |
| 浏览器 CORS 报错 | `digital_human_interface` 与（若直连）Paddle 的允许源未包含前端 Origin |
| 迁机后全挂 | 各服务 `.env` 仍为旧绝对路径或旧 IP |
| 控制台显示服务「已停止」 | `main.py` 用 `127.0.0.1` 探测固定端口；服务未起或端口不一致 |
