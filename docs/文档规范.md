## 2. 文档格式规范

### 2.1 通用格式要求

- **文件格式**：Markdown (.md)
- **编码**：UTF-8
- **命名规范**：
  - 英文文档：使用小写字母，单词间用连字符 `-` 分隔
  - 中文文档：使用中文标题，避免特殊字符
  - 技术文档前缀：可使用 `jna` 前缀标识内部技术文档系列

### 2.2 内容结构

所有文档应包含以下基本结构：

1. **标题**：文档的主标题
2. **摘要**：文档内容的简要说明
3. **目录**：自动生成的目录（如适用）
4. **正文**：详细内容
5. **参考资料**：相关链接和参考
6. **版本信息**：版本号、更新日期、作者

### 2.3 排版规范

- **标题层级**：使用 # 符号表示，最多使用 4 级标题
- **列表**：使用 `-` 或 `*` 表示无序列表，使用数字表示有序列表
- **表格**：使用 Markdown 表格语法
- **代码块**：使用 ``` 包裹，指定语言
- **链接**：使用 `[链接文本](链接地址)` 格式
- **图片**：使用 `![图片描述](图片路径)` 格式
- **强调**：使用 `*斜体*` 或 `**粗体**`

## 4. API 文档规范

### 4.1 接口文档格式

**内容要求**：
- 接口路径
- 请求方法
- 参数说明（类型、必填、默认值）
- 成功返回结构
- 错误返回结构
- 示例代码

**示例格式**：

```markdown
## API 接口：获取视频列表

### 请求信息
- **路径**：`/api/videos`
- **方法**：GET
- **参数**：
  | 参数名 | 类型 | 必填 | 描述 |
  |-------|------|------|------|
  | page | number | 否 | 页码，默认 1 |
  | limit | number | 否 | 每页数量，默认 10 |

### 成功返回
```json
{
  "code": 200,
  "data": {
    "list": [...],
    "total": 100
  },
  "message": "success"
}
```

### 失败返回
```json
{
  "code": 400,
  "message": "参数错误"
}
```
```

### 4.2 API 版本管理

- API 版本应在路径中体现，如 `/api/v1/`
- 版本变更应在文档中记录
- 废弃的 API 应明确标记

## 6. 文档更新机制

### 6.1 更新触发条件

- 代码变更影响功能或接口时
- 架构调整时
- 新增功能时
- 发现文档错误时

### 6.2 更新流程

1. **变更识别**：开发人员识别需要更新的文档
2. **文档修改**：根据代码变更修改相应文档
3. **提交审核**：提交文档修改进行审核
4. **发布更新**：审核通过后发布更新

### 6.3 版本控制

- 文档应与代码版本保持一致
- 重要变更应在文档中记录版本历史
- 使用 Git 进行文档版本管理

## 7. 文档审查和验证流程

### 7.1 审查流程

1. **自我审查**：文档作者进行自我审查
2. **技术审查**：相关技术人员审查技术准确性
3. **内容审查**：审查文档内容的完整性和清晰度
4. **格式审查**：审查文档格式是否符合规范

### 7.2 验证方法

- **技术验证**：通过实际操作验证文档中的技术内容
- **用户验证**：让用户测试文档的可理解性和指导性
- **一致性验证**：验证文档与代码的一致性

### 7.3 审查标准

- 技术准确性：文档内容与实际代码一致
- 完整性：文档包含所有必要信息
- 清晰度：文档表达清晰，易于理解
- 格式规范：文档格式符合规范要求
- 时效性：文档内容与最新版本一致
